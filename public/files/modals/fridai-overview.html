Core Concept

Specs: Automation as Code

Everything in FridAI is driven by a spec file â€” a YAML document that declares exactly what an automation does, step by step.

Think of it like a recipe: each step names a handler (a built-in action), provides its inputs, and optionally flags that it needs human approval or sandboxed execution.

  • Human-readable intent field
  • Ordered task list with typed parameters
  • Per-task approval and sandbox flags
  • AI-searchable via the Memory service
  • Validated before any execution begins
spec:
  id: spec.git_pipeline
  intent: "Create branch, commit changes,
          push for review"

  tasks:
    - id: create_branch
      handler: git_branch
      params:
        name: feature/my-change
        create: true

    - id: commit_work
      handler: git_commit
      params:
        message: "feat: add new feature"

    - id: push_branch
      handler: git_push
      requires_approval: true  # â† human gate
      sandbox:
        required: true
        providers: [docker]

Security

The Docker Sandbox

For high-risk operations, FridAI runs code inside a hardened Docker container. Five phases of hardening went into this â€” it's production-grade isolation.

Default profile locks down

  • Non-root user (uid 1000) â€” never runs as admin
  • All Linux capabilities dropped
  • Network disabled by default
  • Read-only filesystem
  • 512 MB RAM Â· 1 CPU Â· 256 max processes
  • Seccomp filters + AppArmor on Linux

Artifact signing

Sandbox manifests are signed with HMAC-SHA256 and include per-file SHA256 hashes for integrity verification.

Sandbox profiles

Security posture is driven by sandbox profile YAML:

  • default â€” network=none, 512MB/1CPU/256 PIDs
  • dev â€” more permissive for development
  • production â€” locked-down baseline
  • local-dev â€” local development overrides

Defined in packages/system/sandbox/profiles/*.yaml and selected via FRIDAI_PROFILE when sandbox runs

Approvals â€” humans stay in control

Any task can be flagged requires_approval: true. Before that task runs, someone must explicitly sign off â€” via CLI, or through the AI assistant itself.

Each approval record stores:

  • Handler ID and class (SENSITIVE / INFRASTRUCTURE)
  • Who requested it and who approved it
  • Timestamps for full traceability

AI-Assisted Approval Flow

An AI agent can call spec_approval_request over MCP, poll approvals.status/{record_id}, and continue once a human records a decision via spec_approval_record or CLI.

This is what makes FridAI different from just calling APIs directly â€” every sensitive action has a documented human decision attached to it.

MCP Hub Â· Profiles

Profiles â€” one deployment, one posture

A profile (preset) is a YAML file that controls which tools are visible, which handler classes run autonomously, what notifications fire, and how much LLM budget is allowed. The active profile is set server-side via FRIDAI_PROFILE â€” not by the caller.

default

dev / local
  • All handler classes auto-approved â€” fully autonomous execution
  • All MCP tools & resources visible ("*")
  • Notifications: disabled
  • Memory: local JSON
  • Sandbox: Docker (sandbox-base:latest)
  • LLM budget: none

Great for local development where you own the machine and trust all actions.

production

governed / prod
  • Only READ_ONLY handlers auto-approved
  • SENSITIVE & INFRASTRUCTURE require approval by default
  • MCP tools/resources: inherited from base ("*") unless restricted
  • Notifications: Slack + CLI enabled
  • Memory: Pinecone (index fridai-prod)
  • Remotes: github_remote enabled (local remotes off)

Production posture with approvals, notifications, and remote GitHub MCP.

How tool scoping works

Profiles define tool patterns using glob syntax. The hub filters the tool list before the AI even sees it:

# example profile MCP section
enabled_tools:
  - spec_*      # all spec tools
  - runs.*      # run history
  - approvals.* # approval ops

The AI can't call what it can't see.

Optional Plugin

Local LLM support â€” fully offline operation

The optional llm plugin adds a local inference handler backed by Ollama. Run AI models entirely on your own machine â€” no API keys, no data leaving your network.

The llm.generate handler integrates with governance metadata and token budget enforcement, and can be sandboxed when configured.

Recommended models

  • qwen2.5:3b â€” fast, great for tool-calling
  • qwen2.5-coder:7b â€” code generation tasks
  • deepseek-r1:7b-qwen-distill-q4_K_M â€” analysis & reasoning

Token budget management

Each llm.generate call checks the configured token budget (llm.budget_tokens) and fails with BUDGET_EXCEEDED if the call would exceed it.

Plugin auto-discovery

The llm package registers itself via Python entry points. Install the package and it's automatically available as a handler â€” no config changes needed.
